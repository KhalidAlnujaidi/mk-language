Core tensions to resolve before writing code
1. You have two tech stacks and the boundary between them is fuzzy. The broker/hooks/squire CLI is Python (uv, FastAPI, Typer). The business modular monolith is NestJS/TS (or Go). That's fine — they're different problems — but the doc reads as if they're one system. They're not. Make the split explicit:

The ML governance plane (squire + broker + hooks + daemon) is Python. It owns model lifecycle, input grooming, action logging, rail enforcement.
The business application plane (the modular monolith) is NestJS/TS. It owns domain logic, persistence, transport.
The contract between them is the OpenAI-compatible HTTP API + a typed TS SDK. Nothing else. The business app never imports Python; the Python plane never knows about your domain modules.
This is the same Ports & Adapters principle you're already applying inside the monolith — just lift it one level. The capability manifest lives on the Python side and is exposed via /broker/status. The TS SDK reads it and routes accordingly.



2. "Protected rails" via hooks is not protection. The doc acknowledges --dangerously-skip-permissions bypasses hooks, then proceeds as if hooks are the rail. They're a speed bump. Real protection for next.md / alignment/ needs OS-level enforcement:

Linux: chattr +i on protected files, or run the agent under a separate uid with those paths mounted read-only.
macOS: a restricted symlink + SIP-aware path, or run the agent in a separate user account.
Recovery path documented explicitly (boot to recovery, chattr -i, edit, re-lock).
If you're not willing to do OS-level enforcement, be honest in the docs that the rails are advisory, not enforced. The current framing sets the wrong expectation.

3. Self-evolving without an eval harness is just drift. You can't evolve what you can't measure. The EvoAgentX reference is a curated list, not a framework — you'll be building bespoke. Before any self-evolving code ships, you need:

A golden eval set: 20–50 tasks with expected behaviors (not exact outputs — behavioral assertions: "did it redact the secret?", "did it route to the cheap model?", "did it refuse the destructive command?").
A regression runner that executes the eval set against any proposed change (prompt tweak, hook addition, router scoring change).
A versioned artifact store: every evolution proposal is a git branch + eval diff. Merge requires eval pass + human approval for anything touching code; auto-merge only for prompt/config within a sanctioned schema.
Without this, "self-evolving" becomes "the system slowly breaks itself in ways you don't notice." The user's-next-turn-correction signal is excellent — but only if captured structurally (previous output hash, correction type, diff, sentiment) and fed back through the eval harness, not applied directly.

Concrete enhancements
A. Formalize the ground-truth taxonomy. The asymmetry rule is currently aspirational. Make it executable:

TaskKind = DETERMINISTIC | STRUCTURED | FUZZY | REASONING

DETERMINISTIC (regex/AST/git/fs) → no model, plain code.
STRUCTURED (NER, classification, constrained JSON) → small local model with grammar-constrained decoding.
FUZZY (intent, expansion, summarization) → small local model, capped tokens.
REASONING (code generation, review, planning) → broker picks from reasoning-tier models.
The router is then a function of TaskKind + length estimate + capability requirements, not a free-form classifier. This makes routing auditable and testable.

B. Sign the capability manifest. A model declares capabilities; the broker verifies by running a canary task on first registration. Failed canary → capability recorded as null (unknown), not false (verified absent) and never true (assumed). This is your own principle applied to your own registry. Models that hallucinate capability get quarantined.

C. Add an agent outbox. Mirror the transactional outbox for agent actions. Every intended file edit / shell command / tool call is written to a durable log before execution. On crash/hang, the log is the replay source. On success, the log is the audit trail. On user correction, the diff between logged intent and final state is your free quality label. This single pattern gives you: crash recovery, audit, and RLHF signal from the same data structure.

D. Make the warm-set size a function, not a config. warm_set_size = floor(available_vram * 0.6 / max_warm_model_footprint). The 0.6 reserves headroom for the active task's KV cache. Auto-tune the coefficient from observed OOM events (if OOM count > threshold in last N sessions, drop to 0.5). This is a tiny piece of the self-healing layer that pays for itself immediately.

E. One trace, squire to file edit. OpenTelemetry with a single trace context propagated through: squire hook → HTTP → broker → model inference → agent → tool call → file edit. The trace ID is the correlation ID. Don't have two ID systems. Inject at the squire layer (the first touch point), propagate everywhere, export to Jaeger. This is "unified observability" made concrete rather than aspirational.

The NestJS modular monolith is a separate repo (or a separate workspace in the monorepo with its own pyproject/package.json boundary). Mixing ML infrastructure with business application code in one tree creates exactly the coupling you're trying to prevent.

G. The scaffold CLI is the highest-leverage piece. squire make:module ordering generating the full clean-architecture skeleton + registering it + emitting test stubs is worth more than any amount of documentation. Build it first. If developers can't scaffold a new module in 10 seconds, the architecture won't survive contact with deadlines.

H. Drizzle over Prisma. Correct preference. Drizzle's schema-per-module approach fits modular monoliths; Prisma's single-schema generator fights you. Also: Drizzle's query builder composes, Prisma's API doesn't — and in a modular monolith you'll be composing queries across module boundaries via ports.

EvoAgentX population methods are more optimization/search-driven than classic genetic-style population evolution, but they draw heavily from evolutionary computation principles.
EvoAgentX (the framework) integrates three key MAS optimization algorithms in its evolving layer for iterative refinement of prompts, tools, and workflow topologies:
1. TextGrad (Gradient-like Textual Optimization)

Treats prompts/responses as "differentiable" via natural language feedback ("text gradients").
Iteratively refines components (e.g., agent prompts) based on evaluation feedback, similar to gradient descent but in text space.
Strong for prompt engineering and chain refinement; efficient for targeted improvements without full population sampling.


AFlow (Reinforcement Learning-inspired with Monte Carlo Tree Search)

Models workflow evolution as a search problem over graph topologies or execution paths.
Uses MCTS-style exploration to discover better agent collaboration structures, with selection/mutation-like steps.
Balances exploration (new workflow variants) and exploitation (proven good ones).

3. MIPRO (Model-agnostic Iterative Prompt Optimization)

Black-box optimization focused on prompts and configurations.
Uses adaptive reranking and iterative proposals based on performance evaluations.
Efficient for tuning without needing gradients or full retraining.

These are applied in loops: generate/evaluate → optimize (via one or more of the above) → update workflow. The framework supports automatic workflow generation from high-level goals, then evolutionary refinement, with built-in evaluation (benchmarks like HotPotQA, MBPP, MATH, GAIA).


Broader Context: Population-Based Methods in Self-Evolving Agents
EvoAgentX is part of a larger ecosystem (see their Awesome-Self-Evolving-Agents repo). True population-based evolutionary algorithms appear more prominently in related works:

EvoAgent (distinct but related): Explicitly uses evolutionary algorithms to extend single agents into multi-agent systems. It maintains a population of agent variants (different roles, prompts, tools), applies operators like mutation (slight variations), crossover (combining successful traits), and selection (fitness-based, e.g., task performance). Starts from an initial "individual" (existing agent/framework) and evolves generations. Population size and iterations are tunable (e.g., GROUP_NUM). This enables automatic specialization and collaboration.
General Evolutionary Patterns (from surveys):
Generate a population of candidate agents/workflows (genomes encode prompts, topologies, tools, memory).
Evaluate fitness on tasks/datasets (accuracy, efficiency, robustness).
Select top performers (tournament, elitism).
Vary via mutation/crossover to create next generation.
Constraints: population size (for cost), diversity maintenance, evaluation budget.


This mirrors classic Genetic Algorithms (GA), Evolutionary Strategies, or neuroevolution, but adapted to LLM agents (textual "genomes," LLM-driven mutation, black-box fitness via execution).
Relevance to Kinox/Kniox Broker & Self-Evolving Design
This is highly applicable and not too visionary for your local broker daemon + meta-review loop:

Population in Broker: Maintain a small, persistent "population" of model profiles/configs (e.g., different grooming/review prompts, routing heuristics, hook variants). Use cheap local models for fitness evaluation on historical logs/tasks.
Evolve Routing/Grooming: Apply MIPRO/TextGrad-style iteration or light EvoAgent-style population (e.g., 5-10 variants) for router scoring or preprocessor stages. Selection based on metrics (latency, secrets caught, review quality).
Workflow Evolution: For agent sessions or reviewers, evolve topologies (e.g., add/remove sub-agents) using AFlow-like search.
Self-Healing Tie-in: Use evolution to discover recovery strategies (e.g., auto-fallback configs) from failure logs.
Practical Implementation:
Start small: Population size 4-8, 2-5 generations per meta-cycle (nightly or post-session).
Fitness: Multi-objective (performance + cost + safety alignment with next.md).
Human gate: Propose top variants for review/merge into registry/hooks.
Local-first: Run on cheap models; use your broker for inference.


Pros: Leads to genuine adaptation (e.g., project-specific optimizations). Cons/Risks: Evaluation cost (mitigate with cached logs + cheap evaluators), diversity collapse (add explicit diversity metrics), safety (guard evolved code/prompts heavily).

## The Core Architecture

The ultimate foundation here is a Modular Monolith using Clean Architecture principles. You want to isolate your core business logic entirely from your transport layers (HTTP/gRPC) and data layers (Databases/APIs).

Strict Domain Isolation: Your business rules shouldn't care if they are triggered by a REST controller, a Kafka message handler, or a local CLI script. Keep the core domain logic "pure."

Ports & Adapters (Hexagonal Architecture): Define explicit interfaces (Ports) for database access, third-party logistics integrations, or notification engines. The concrete implementations (Adapters) live on the outer edges. This makes switching a database or mocking an external service for testing trivial.

Logical Modules over Microservices: Don't split into separate microservices on day one. Instead, build highly isolated modules within the same repository that only communicate via public interfaces or an in-memory event bus. This keeps your deployment dead-simple while giving you a clear path to slice out a microservice later if performance demands it.

## High-Scale & Reliability Engine

Operating at scale requires defensive design patterns. Your new foundation should have infrastructure guardrails baked directly into the template.

Transactional Outbox Pattern: Never save data to your database and publish an event to a message broker (like RabbitMQ or Kafka) in the same breath. If the broker is down, your system becomes inconsistent. Instead, write the event to an outbox table in the same database transaction, and let a background worker process it safely.

Idempotency Handlers: In transactional systems, network glitches cause clients to retry requests. Build an inline idempotency mechanism (ensuring a duplicate request doesn't trigger duplicate actions) using a Redis middleware or database constraints based on a unique request ID.

Unified Observability: Don't treat logging as an afterthought. Bake OpenTelemetry into the core bootstrap layer. Every log, trace, and metric should carry a unified correlation ID from the moment it enters the system, allowing you to trace a request's entire journey across modules.

The three are one idea at three altitudes

kinox — the model as the system. "The model is the shell; the screen is render-only generative output," a bootable appliance with a baked-in local LLM. Altitude: the whole machine.
kniox — governing the agent on the system. The agent is the guest; fail-closed hooks, a VRAM broker, dispatch ("one model per task"), an opt-in Tailscale fleet. Altitude: the agent's actions and resources, in userspace.

You've been circling a convergence point.
The best of each — the pieces that survived all three rewrites
This is the real signal. The same six primitives keep reappearing, each time more battle-tested. These are your kernel:

Capability manifest — detect, never assume; null, never a fabricated zero. (kniox's probe, kinox's model_select.) Born from a real bug — the best kind of primitive.
Asymmetric router — ground-truth/deterministic vs fuzzy/model, and local-rung vs cloud-tier. Squire states the law most cleanly: if there's a ground truth, plain code beats the model. kniox's dispatch and kinox's ladder are the same router at different scopes.
Policy gate with an explicit fail-direction — kinox's propose→confirm→cancel plus conservative is_affirmative, kniox's fail-closed hooks, squire's fail-soft laws. The insight hiding here: fail-direction is per-component (a guard fails closed, an optimizer fails soft), so it should be a first-class property of the gate, not a global default.
Execution boundary — kinox's bwrap recipe: a pure argv builder (unit-testable policy) + a capability probe + actionable hints. The cleanest sandbox of the three.
Working memory + corpus — next.md / CLAUDE.md / boundaries.md, the brainstorm→spec→plan→TDD flow, and the append-only, secret-scrubbed session log both roadmaps independently reinvented.
Honest observability — exact where you can measure, labeled estimates where you can't (squire's token ledger), plus the highest-leverage idea in either roadmap: the user's next-turn correction is a free quality label.

Self-Healing Mechanisms
Focus on automatic detection + recovery for common failures (crashes, OOM, model unload, hook errors, config drift, hardware changes, session hangs).
1. Broker Daemon Self-Healing (Core)

Health Supervisor: Watchdog process (systemd or supervisord) + internal heartbeat. On failure: auto-restart with exponential backoff. Graceful shutdown/unload of models first.
Resource Guardian:
Continuous VRAM/CPU monitoring.
On OOM risk: Auto-unload least-used models, downgrade quant, spill to CPU, or queue/reject low-priority tasks.
Post-failure: Analyze logs, suggest model registry tweaks (e.g., "reduce concurrent for gemma3").

Task Recovery:
Retry policy per task type (grooming: 3x fast; deep review: human escalate).
Circuit breakers: If a backend (Ollama) flakes, fallback chain + temporary quarantine.
Checkpointing: For long sessions, save partial state (context, diffs) to resume.

Config & State Reconciliation:
Periodic kx doctor --auto-fix: Validate hardware manifest vs. runtime, reset drifted TOML, pull missing models.

Hook/Preprocessor Resilience:
Wrap all hooks in try/except with sandboxed execution (subprocess limits).
On failure: Fallback to deterministic rules or cheaper model; log for review loop.


2. System-Wide Healing

Session Guardian: kx launcher monitors agent sessions. On hang/crash: Snapshot, rollback protected files (next.md, alignment), notify + offer replay with different model.
Log-Driven Auto-Remediation: Dedicated reviewer agent (cheap local) scans error patterns → generates patches for hooks/preprocessors/broker. Human PR approval (or auto for low-risk).
Hardware Adaptivity: On GPU change/hotplug: Re-run detection, update registry, restart affected models.

Implementation Tip: Use libraries like tenacity for retries, psutil/pynvml for monitoring, and a simple state machine in the daemon.
Self-Evolving Mechanisms
Make the system learn from usage to improve prompts, hooks, routing, models, etc. — without full autonomy (keep governance).
Core Loop (Meta-Review Cycle):

Observe: Aggregate structured logs/metrics (tasks, models used, latency, success rate, user interventions, diffs applied).
Analyze: Periodic (post-session or nightly) cheap local reviewer + frontier model (capped) evaluates:
Failure patterns.
Performance (e.g., "groomer too verbose → shorten").
Opportunity (new common intent → specialize model).

Propose & Apply:
Generate improvements: Updated prompt templates, new preprocessor stage, router scoring tweak, hook addition.
Store in evolutions/ or Git-tracked improvements/.
A/B test safely (shadow mode) or staged rollout.
Human review gate for code changes; auto-apply for prompts/config.

Validate: Run regression tests/smoke + metrics before merge.

Specific Evolution Targets:

Broker/Routing: Meta-agent evolves model profiles, scoring heuristics, or adds new backends based on hardware/logs.
Preprocessors & Hooks: Auto-refine stages (e.g., better redact regex from caught secrets), suggest new guards from failure modes.
Model Registry: Suggest fine-tune jobs (via LoRA on usage data) or new Ollama pulls. Track "model fitness" scores.
Project Scaffolding & Alignment: Evolve templates/next.md structures from successful projects.
Skills/Hooks: Agent proposes new single-file skills based on repeated user patterns.

Advanced (Kinox Vision):

Self-Referential Editing: Broker or dedicated "architect" agent can propose edits to its own codebase (with heavy guards: diff review, tests, git revert easy).
Population/Evo Methods: Maintain a small set of prompt variants or hook versions; evolve via selection (inspired by Darwin-Gödel or EvoAgentX).
Memory: Persistent vector store (local) of past evolutions + outcomes for better proposals.

Scope Limits: Start with config/prompts/hooks; code edits are high-risk (sandboxed or PR-only).

Cost: Mostly local models for meta-work.

The Cross-Stack Shared Infrastructure (The Bedrock)
Regardless of whether you choose Go or TypeScript, the infrastructure supporting your application needs to remain identical to guarantee production resilience:

Primary Database: PostgreSQL (supports robust ACID transactions necessary for the Outbox pattern, and has excellent JSONB indexing features).

Caching & Idempotency: Redis (used as a distributed lock manager and high-speed temporary storage for processing unique token requests).

Local Playground Orchestration: A standardized docker-compose.yml spinning up PostgreSQL, Redis, and an observability stack (Jaeger for tracing, Prometheus for metrics).

Architectural Decision Matrix
Go is the choice if your primary bottlenecks are CPU, memory consumption, or ultra-low latency response requirements under intense loads.

TypeScript/NestJS is the choice if your bottleneck is shipping features quickly, onboarding developers fast, and managing complex polymorphic data structures with minimal code footprint.

1. Layer 1 (Input Grooming) Enhancements
Exact & Semantic Local Caching Combo
Running a local model (like Gemma or Qwen) for every prompt expansion or intent classification introduces unnecessary millisecond delays.

The Cheap Fix: Implement a two-tiered local cache using an append-only SQLite database in WAL (Write-Ahead Logging) mode.

Tier 1: An exact MD5 hash match of the incoming raw prompt. If matched, instantly return the previously generated intent/tags.

Tier 2: A lightweight, localized semantic cache using raw cosine similarity over cheap local embeddings (e.g., using a tiny 33MB BGE-micro model). If the user asks for variations of the same routine task, bypass the local LLM entirely.

Git-Diff Context Pruning
Agents often hallucinate or drown in context when given entire files that haven't changed.

The Cheap Fix: Instead of feeding full files to the prompt grooming layer, use a native regex/git shell wrapper to pull only the active git diff and the corresponding structural symbols (class signatures, function headers) using ctags. This costs virtually zero compute, slashes your token footprint, and keeps the agent hyper-focused on the exact surface area of the current change.

2. Layer 2 (Runtime Governance) Enhancements
Unix Domain Sockets (UDS) over TCP for the Daemon
If the kx CLI tool, your preprocessor hooks, and your backend app communicate with the Model Control Plane (mcp) daemon over local TCP loops (localhost:11435), you incur network stack serialization overhead.

The Cheap Fix: Bind the daemon to a native Unix Domain Socket (e.g., /var/run/kinox.sock). IPC (Inter-Process Communication) over a socket bypasses the network loopback entirely, dropping latency to near-zero and hardening the system against local port collisions or unauthorized local network scanning.

The "Passive Correction" Data Harvester
Your roadmap notes that the user's next-turn correction is a free quality label. Let's make collecting it passive and automatic.

The Cheap Fix: When an agent writes code or edits a file, the kx launcher takes a snapshot of the file state. If the human immediately opens that file and manually edits the exact lines the agent touched within a 2-minute window, a post-commit hook packages the Agent's Output vs. Human's Correction into a local JSONL file (~/.kinox/evals.jsonl). You are building a production-grade, self-hosted evaluation dataset for fine-tuning future local models completely for free.

3. Modular Monolith & Infrastructure Enhancements
Zero-Infra SQLite Outbox for Local Dev
While your production system uses PostgreSQL for the Transactional Outbox pattern, spinning up a full Postgres instance during local scaffolding or isolated testing can slow down the developer lifecycle.

The Cheap Fix: In your local environment or test suites, swap the Outbox storage to an in-memory or single-file SQLite database. Inline Token Budgeting & Early Exits

The Cheap Fix: Build a primitive token-counter middleware directly into the kx launcher wrapper. Two more cheap wins from forgotten corners
Plan 9's per-process namespaces are your context-pruning done structurally instead of by filtering. Rather than deciding what to remove from the agent's view, you mount only what it should see — the job's namespace literally doesn't contain the files it has no business touching. Union mounts and 9P are the mechanism; the idea is that context control becomes a property of the constructed namespace, not a hook that strips things after the fact. And content-addressed code (Nix's purely-functional, reproducible builds; Unison's code-identified-by-hash-of-its-AST) generalizes the MD5 prompt cache you already have: content-address your stages, prompts, and manifests and you get perfect caching and reproducibility with cryptographic guarantees, and "it worked on my machine" largely dies.
If I had to rank by ROI
Tightly, highest leverage first: (1) define the Job/Process object — it unlocks everything else; (2) formalize the syscall table + adopt the ocap mental model — your security story in one chokepoint using the manifest you already have; (3) supervision trees (steal OTP's shape) — gets self-healing as structure, not patchwork; (4) policy-as-logic via something Rego-shaped — explainable, evolvable gate; (5) the executor-utilization metric (frontier-tokens per unit work) as the north star; (6) structured concurrency for subagent budgets; (7) lottery/proportional-share scheduling for the broker. Everything below that — single-level store, Plan 9 namespaces, content-addressing, tuple spaces — is real value but a tier down in urgency.
The throughline: you're not missing features, you're missing named primitives. You've reinvented half a dozen OS concepts under ad-hoc names, and the cheapest gain available is to recognize which classic primitive each one is, then steal its mature design instead of evolving your own.
